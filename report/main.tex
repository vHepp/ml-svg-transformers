\section{Introduction}
% (0.5–1 page)

\subsection{Motivation}
% Motivation for studying scaling laws on SVG data

Vector graphics are fundamentally different from rasterized pixel images;
they are resolution-independent and defined entirely by mathematical paths and shapes described in XML.
This makes Scalable Vector Graphics (SVGs) an excellent testbed for sequence modeling.
By training a Transformer on SVG code, we can investigate whether language models
can learn rigid syntactical grammar, mathematical coordinate structures, and continuous geometric
relationships simultaneously---where the output can be immediately verified through visual rendering.
% \todo{de-geminify}


\subsection{Approach}
% Overview of your approach

We approach this problem by training a family of decoder-only Transformer language models of varying
widths from scratch on a curated dataset of SVG icons. To analyze the scaling behavior of these models,
we compare Standard Parameterization (SP) against Maximal Update Parameterization (\mup).

By establishing an empirical scaling law, we aim to demonstrate how \mup~enables zero-shot hyperparameter transfer,
allowing us to find an optimal learning rate on a proxy model and safely scale it up to our largest architecture.
Finally, we evaluate the generative capabilities of the fully trained models through conditional and unconditional SVG rendering.

\subsection{Dataset and Preprocessing}
% Dataset description and preprocessing pipeline


The training data was sourced from the \verb|starvector/svg-stack-simple| dataset via Hugging Face~\cite{rodriguez2023starvector}.
To normalize the data, we applied a strict regex-based cleaning pipeline.
All XML comments and metadata were stripped, and coordinate precision was rounded to a single decimal place.
The string data within the \verb|<path>| tags was spaced out to isolate individual command letters (i.e., M, L, C)
from numerical coordinates. Finally, any SVGs containing fewer than 50 characters or exceeding \verb|TOKEN_THRESHOLD|
of 2,048 characters were filtered out. This threshold was chosen to balance the complexity of the SVGs with the
memory constraints of our \verb|BLOCK_SIZE = 256| context window, ensuring the self-attention mechanism remained computationally feasible.
We also included checks for broken XML encodings and SVG rendering errors, but we never experienced these during data cleaning.

Initially, we worked with the \verb|starvector/svg-icons-simple| and \verb|starvector/svg-emoji-simple|
datasets, but their combination alone yielded fewer than 100 million tokens.
Figure \ref{starvec_table} shows a size comparison of the three datasets.

\section{Methods}
% (1.5 - 2 pages)

\subsection{Model architectures}

Our CustomTransformer model class shown in figures \ref{Head}--\ref{CustomTransformer} was built from the ground up with PyTorch
following the original \textit{Attention Is All You Need} paper and Andrej Karpathy's \textit{nanoGPT} repository \cite{Vaswani,nanoGPT}.
Five configurations of the model were considered for our experiments. Due to \mup~package requirements,
the depth dimensions (n\_layers and n\_heads) were held constant across all models while width dimensions
(d\_model and d\_ff) varied. The model configurations are shown in figure \ref{model_configs}.

\subsection{Training setup and hyperparameters}

The models were trained using the AdamW optimizer paired with a cosine annealing learning rate scheduler with a linear warmup period
spanning the first 5\% of training steps. During our full training of the \mup-tuned ``XL'' model, we swapped out our manual attention calculations
with PyTorch's native \verb|F.scaled_dot_product_attention| (FlashAttention) to speed up our iteration throughput.
We utilized three unique platforms during this project: An M4 Macbook Air with MPS acceleration, a Windows 10 PC with NVIDIA RTX 3080 GPU + CUDA,
and NYU HPC's Google Cloud Bursting with an NVIDIA A100 GPU + CUDA.


\subsection{Experimental design for scaling studies}

To analyze the scaling behavior of the Transformer, we evaluated the models across the
5 varying parameter counts outlined in Figure \ref{model_configs}. The depth of the network
(number of layers \verb|n_layers| and attention heads \verb|n_heads|) was kept constant,
while the width (embedding dimension \verb|d_model| and feed-forward dimension \verb|d_ff|) was scaled.
We compared the performance of models using Standard Parameterization (SP) versus Maximal
Update Parameterization (\mup). To do this, we first performed a 1000-step learning rate sweep on
the proxy ``Tiny'' model to identify the optimal base learning rate. 7 orders of learning rates were
tested: LR = $10^{-7}$, $10^{-6}$, $10^{-5}$, $10^{-4}$, $10^{-3}$, $10^{-2}$, and $10^{-1}$.
Once the optimal learning rate was identified, we then trained each of the 5 models on one full pass of
our training data subset, tracking both training and validation losses along the way.
Using the \mup~framework, we repeated this process again to find a potentially new optimal learning rate
and apply it across all larger model sizes to observe zero-shot hyperparameter transfer and
plot the resulting training and validation losses.

\subsection{Evaluation metrics}

The primary quantitative metric for model performance during training was the Cross-Entropy
Loss on the validation dataset, representing the model's ability to accurately predict the
next token in the SVG sequence.

For qualitative evaluation, we assessed the generated SVGs based on two structural criteria:
\textit{XML validity} (whether the generated string could be parsed correctly as a valid XML
document using the \verb|lxml| library) and \textit{renderability} (whether the string could
be successfully rasterized into a PNG using CairoSVG). We evaluated these metrics across
batches of both unconditional and conditional generation prompts.


\section{Results}

\subsection{Fixed LR vs. \mup~comparisons and scaling}


Figure \ref{FixedLR} shows that the standard fixed learning rate approach ($1 \times 10^{-2}$) works fine while training smaller models,
but not for the largest models we tested. Almost immediately into training 1 epoch, the ``Large'' and ``XL''
model's training loss curves diverge from the smaller models. We also see that validation loss
does not follow a power-law relationship with respect to the model parameters, as the fitted scaling
exponent $\alpha = 0$.

In contrast, figure \ref{muLR} shows that the \mup~framework stabilized the optimal learning rate on the
``Tiny'' model and successfully transferred it to the ``Large'' and ``XL'' models without causing training divergence.
The validation loss follows a power-law relationship with respect to the model parameters, as the fitted scaling
exponent $\alpha \approx -1.83$.

\subsection{Scaling law extrapolation prediction}

When considering a model with 440M parameters---10x the parameters of our ``XL'' model---that is tuned with \mup,
we would confidently predict the training loss to be quite similar at around 0.46. This prediction follows directly from the fitted
power law in figure \ref{muLR}.

\subsection{Generated samples with evaluation}

Our fully trained XL model was evaluated by generating 10 unconditional samples and 5 conditional
samples (Figures \ref{Unconditioned} and \ref{Conditioned}). Using temperature scaling ($0.7$) and
Top-K filtering ($K=5$), the model achieved a 100\% success rate in both XML validity
(10/10 unconditional, 5/5 conditional) and renderability. However, the model would occasionally
try to generate strings longer than our \verb|max_new_tokens = TOKEN_THRESHOLD| limit while testing.


While the model successfully learned the strict syntactic grammar of SVGs, a post-processing function
was required to resolve spacing artifacts introduced by the tokenizer. After post-processing, the rendered
outputs visually resembled abstract geometric squiggles rather than coherent icons.

\section{Discussion}
% (1-2 pages)

\subsection{Key insights from scaling experiments}

The scaling plots in figure \ref{muLR} confirm that increasing model width reliably improves the next-token
prediction capabilities on SVG data, following predictable power-law dynamics, if proper care is taken when
choosing a learning rate based on model size.

The results strongly validate the Maximal Update Parameterization framework. Under Standard
Parameterization, the optimal learning rate shifts as model width increases, making
hyperparameter tuning for large models prohibitively expensive. However, \mup~allowed us to
tune the learning rate cheaply on the 1.5M parameter ``Tiny'' model and safely transfer it to the
44M parameter ``XL'' model. The \mup~models maintained stability at learning rates that would
typically cause divergence in standard Transformers.


\subsection{Design decisions and their impact}

Several design choices significantly impacted our training pipeline. Processing
the HuggingFace datasets via PyArrow's zero-copy memory mapping prevented the CPU from
bottlenecking the GPU data loaders. Furthermore, implementing FlashAttention avoided
materializing the $N \times N$ attention matrix in memory, which nearly
quadrupled our iteration speed and saturated the A100 GPU's memory bandwidth.

\subsection{Limitations and future work}

The \mup~framework required us to hold depth dimensions constant across all the models we tested, so our results
are comparable when varying the width dimensions. We realized this limitation late into testing, leaving us to apply
a quick fix by holding the depth dimensions constant without modifying the width dimensions. This decision resulted in model
configurations that did not cover as wide of range of parameter counts as originally planned.

Also, a bug in our code caused us to overwrite our final ``XL'' model training loss graphs with the single latest epoch
loss. The plot is shown in figure \ref{last_loss}, but is effectively useless.

\section{Conclusion}
In this project, we successfully collected, cleaned, and tokenized a dataset of over 200,000
SVG paths to train a custom Transformer architecture from scratch. We verified empirical
scaling laws and demonstrated the effectiveness of Maximal Update Parameterization for
zero-shot hyperparameter transfer. While generating meaningful SVG art with
language models remains a challenge, our
models successfully learned to natively write and format functional, renderable SVG files.

		self.key = nn.Linear(d_model, head_size, bias=False)
		self.query = nn.Linear(d_model, head_size, bias=False)
		self.value = nn.Linear(d_model, head_size, bias=False)
		self.head_size = head_size

		# 'tril' is the lower triangular matrix for masking
		self.register_buffer("tril", torch.tril(torch.ones(block_size, block_size)))

	def forward(self, x):
		B, T, C = x.shape  # Batch, sequence length, embedding dimensionality (d_model)

		k = self.key(x)
		q = self.query(x)
		v = self.value(x)

		# Compute attention scores
		wei = q @ k.transpose(-2, -1) * (self.head_size**-0.5)

		# Mask out future tokens by replacing 0s in the tril matrix with -infinity
		wei = wei.masked_fill(self.tril[:T, :T] == 0, float("-inf"))

		output = F.softmax(wei, dim=-1) @ v

		return output
	\end{minted}
	\caption{Head Class}
	\label{Head}
\end{figure}

class MultiHeadAttention(nn.Module):
    def __init__(self, n_heads, d_model, block_size):
        super().__init__()

        head_size = d_model // n_heads

        self.heads = nn.ModuleList(
            [Head(head_size, d_model, block_size) for _ in range(n_heads)]
        )

        self.proj = nn.Linear(head_size * n_heads, d_model)

    def forward(self, x):

        # Runs input through each attention head, and glues them together side-by-side
        x = torch.concat([head(x) for head in self.heads], dim=-1)

        # Mixes the (head_size x n_heads) head outputs back to one d_model dimensional layer
        x = self.proj(x)

        return x

class TransformerBlock(nn.Module):
    def __init__(self, n_heads, d_model, d_ff, block_size):
        super().__init__()

        self.MHA = MultiHeadAttention(
            n_heads=n_heads, d_model=d_model, block_size=block_size
        )
        self.ff = FeedForward(d_model=d_model, d_ff=d_ff)
        self.ln_1 = nn.LayerNorm(d_model)
        self.ln_2 = nn.LayerNorm(d_model)

    def forward(self, x):

        x = x + self.MHA(self.ln_1(x))
        x = x + self.ff(self.ln_2(x))

        return x

class CustomTransformer(nn.Module):
    def __init__(self, vocab_size, block_size, n_layers, n_heads, d_model, d_ff):
        super().__init__()
        self.vocab_size = vocab_size
        self.block_size = block_size
        self.n_layers = n_layers
        self.n_heads = n_heads
        self.d_model = d_model
        self.d_ff = d_ff
        self.token_embedding = nn.Embedding(vocab_size, d_model)
        self.positional_embedding = nn.Embedding(block_size, d_model)
        self.blocks = nn.Sequential(
            *[TransformerBlock(n_heads, d_model, d_ff, block_size) for _ in range(n_layers)]
        )
        self.ln_f = nn.LayerNorm(d_model)
        self.proj = nn.Linear(d_model, vocab_size)

    def generate(self, idx, max_new_tokens, temperature=0.8):
        for _ in range(max_new_tokens):
            # Crop to block_size if too long
            if idx.size(1) <= self.block_size:
                idx_continue = idx
            else:
                idx_continue = idx[:, -self.block_size :]
            logits, _ = self(idx_continue)
            logits = logits[:, -1, :] / temperature
            probs = F.softmax(logits, dim=-1)
            idx_next = torch.multinomial(probs, num_samples=1)
            idx = torch.concat((idx, idx_next), dim=1)
        return idx

    def forward(self, idx, targets=None):
        sequence_length = idx.shape[1]
        te = self.token_embedding(idx)
        pe = self.positional_embedding(torch.arange(sequence_length, device=idx.device))
        x = te + pe  # input_embedding
        x = self.blocks(x)
        x = self.ln_f(x)
        logits = self.proj(x)
        if targets is None:
            loss = None
        else:
            B, T, C = logits.shape
            logits = logits.view(B * T, C)
            targets = targets.view(B * T)
            loss = F.cross_entropy(logits, targets)
        return logits, loss
